Estimation of chronological age from methylation data¶

Introduction¶

A general introduction to the topic can be found in these slides. However, for those who wish to understand the notebook in greater detail, the original Horvath article (2013) might be a useful reference.

Goal of the analysis¶

The main objective of our analysis is to generate a relatively simple linear regression model which can predict a person's chronological age by observing the levels of DNA methylation in various genomic positions.

Importing necessary python modules¶

The following analysis makes use of only five very basic python modules:

  • gzip for dealing with gzipped files,
  • pandas for manipulating tables and datafiles,
  • numpy for basic mathematical calculations,
  • matplotlib for plotting,
  • sklearn for various linear regression models and related functions.

Downloading and preparing data files¶

Note: If you are short on time, you can skip this step and continue with the next phase of data analysis. However, in general, it is very important to have a good understanding of the data sources you are planning to investigate.

Finding publicly available methylation data¶

One of the most widely used resources for publicly available methylation (and gene expression) data is NCBI's Gene Expression Omnibus (GEO) database. It takes a little practice to get familiar with the system, but the webpage allows you to search for epigenetic datasets by applying multiple filters.

In our case, besides obtaining the DNA methylation levels ($\beta$-values) for the patients in different genomic positions, it is also essential that the true chronological age of each person is available.

To this end, we will use the data from study GSE41037. According to the GEO overview page of the dataset, it contains the blood samples of 720 patients (with or without diagnosed schizophrenia) and methylation levels were determined on the Illumina Infinium 27k Human Methylation Beadchip measurement platform. This means that we have methylation $\beta$-values available in around 27 000 genomic positions for each patient.

Downloading the data file¶

It is somewhat tricky to decide which datafile to download from the GEO website to obtain the most useful dataset. In general, whenever available, the "Series matrix files" of the studies contain the necessary information in the most condensed manner. For our specific case, this file can be downloaded from here with the command below (on a Linux system):

!wget https://ftp.ncbi.nlm.nih.gov/geo/series/GSE41nnn/GSE41037/matrix/GSE41037_series_matrix.txt.gz

Or alternatively, on Windows, you can just click on the download link and download the file to your computer.

# defining function to look into specific lines of a file
def head(filename, minrow = 0, maxrow=10):
    if ".gz" in filename:
        with gzip.open(filename, 'r') as f:
            lines = [f.readline().decode("utf-8") for line in range(1, maxrow+1)]
    else:
        with open(filename, 'r') as f:
            lines = [f.readline() for line in range(1, maxrow+1)]
    return lines[minrow:]
        
# defining function to search for specific strings in a file
def grep(filename, strings = []):
    if ".gz" in filename:
        with gzip.open(filename, 'r') as f:
            lines = []
            for l in f.readlines():
                if len(strings) == 0 or sum([s in l.decode("utf-8") for s in strings]) == len(strings):
                    lines.append(l.decode("utf-8"))
    else:
        with open(filename, 'r') as f:
            lines = []
            for l in f.readlines():
                if len(strings) == 0 or sum([s in l for s in strings]) == len(strings):
                    lines.append(l)
    return lines

Saving condensed datatable for later use¶

It is practical to now save our datatable in a straightforward format, so that we don't have to rerun metadata extraction if we wish to work on it later on.

First we transpose the data matrix and then add appropriate metadata to additional columns.

In [12]:
# setting the index to be probe IDs
methylation_data.index = methylation_data["ID_REF"]
# dropping the probe ID column as it's already present as the index
methylation_data = methylation_data[methylation_data.columns[1:]]

# transposing the dataframe, so each row now corresponds with a sample and each column with a probe
# with measurement beta-values in each cell
methylation_data = methylation_data.transpose()

# adding the chronological age as an additional column
methylation_data["age"] = [age[GEO_ids.index(k)] for k in methylation_data.index]

# adding gender as an additional column
methylation_data["gender"] = [gender[GEO_ids.index(k)] for k in methylation_data.index]

# finally saving the resulting datatable for later use
methylation_data.to_csv("clean_data.csv")

Age transformation¶

To understand the detailed reasoning behind this step, consult the original Horvath article (2013).

Given that "DNA methylation age" is not a linear function of chronological age, but changes more rapidly below around 20 years and slowly saturates after that, it is reasonable to perform a simple (and invertible) transformation of age prior to model fitting. Thus, the target variable in our following analyses is not simply the chronological age, but its transformed version.

The following two functions can be used to transform between the two variables:

In [21]:
def transform_age(a, adult_age = 20):
    x = (a+1)/(1+adult_age)
    return np.where(x <= 1, np.log(x), x-1)
    
def retransform_age(x, adult_age = 20):
    return np.where(x < 0, (1+adult_age)*np.exp(x)-1, (1+adult_age)*x+adult_age)

Preparations for model fitting¶

Train-test split¶

Whenever you wish to fit models to a dataset, it is useful to separate a part of it which can be later used for performance assessment and is somewhat independent of the data that you use for model fitting. However, always keep in mind that the true test of a model has to be performed on a completely unrelated dataset. It is possible that experimental and technical artifacts are highly correlated in a dataset measured at the same time, on the same instrument, with the same pipeline. This means that even if you separate a part of it for later testing, some information about these samples inherently "seep through" to the training set and the model learns these artifacts instead of real trends. This would result in very high performance metrics on the training and test sets but very low on an independent validation set.

Standardization of methylation data¶

It is a general practice in machine learning tasks to rescale the input parameters in a way that "they are on the same scale". For example, if you want to include both the height of a person (measured in cm) and the thickness of their nail (measured in mm) in the same model, the effect of height would "overpower" the effect of nail thickness in a numerical model simply because its values have an average around 160-170 (cm), while nail thickness is around 0.4-0.5 (mm).

In our case, methylation $\beta$-values are relatively similar in the sense that they all fall into the (0,1) range. However, it is still useful to perform the rescaling prior to model building.

One of the most widely used rescaling techniques is "standardization", which calculates the $\mu$ mean and $\sigma$ standard deviation for each input parameter and transforms the values with the following formula:

$$\tilde{x} = \frac{x-\mu}{\sigma}$$

This results in transformed variables with zero mean and a standard deviation of 1.

Note: standardization is first performed on the training set and the transformed variables are used for model building. However, when you test your model on a test set or an unrelated validation set, be sure to standardize their input parameters based on the original training set and not on themselves.

ElasticNet regression - linear regression with L1 and L2-regularization¶

To compensate for overfitting, we add the L1 and L2-norm of the coefficient vector to the objective function with different multipliers and try to find the optimal value of these hyperparameters by minimizing the objective function using cross-validation.

During cross-validation, additional train-test splits and model fittings are performed multiple times on the training dataset to get statistics on performance metrics. Finally, those hyperparameters are chosen that resulted in the "best" model. (Model selection is not a trivial question, you can learn about the approach in a specific setting here.)

def get_horvath_prediction(r):
    pred = 0
    for pi, probe in enumerate(list(horvath_model_coeffs["CpGmarker"][1:])): # skipping intercept
        if pd.isnull(r[probe]): # if no methylation value is present for the probe, treat it as zero
            continue
        pred += r[probe]*horvath_model_coeffs["CoefficientTraining"][pi+1]
    pred += horvath_model_coeffs["CoefficientTraining"][0] # now adding the intercept
    return float(retransform_age(pred)) # transforming prediction

Can we do even better?¶

The frequently overlooked solution for improving model performance is collecting even more training data. This aspect is usually overlooked for good reason, because it generally takes:

  • a lot of time,
  • a lot of money,
  • and a lot of boring and hard work.

In many cases, it is simply not feasible to increase the training dataset and in many cases it's just too tiresome. However, if you plan to design a model that can be used widely, it is worth investing your energy into finding as much data as you possibly can.

On the other hand, additional dataset #2 had patients of ages 49-74 years, so this might give us a slight advantage.

Keep in mind that we still do not really know anything about the possible technical artifacts and batch effects that could influence the results.

Using a validation cohort from a single study is also unwise. It is possible that the given study has some special technical bias that results in poor predictions. It is always good practice to use both the largest possible training set and the largest possible validation set that you can find.

It depends on the given task whether you want to build a model that generalizes well to all imaginable cases (all tissue types, all measurement platforms, all age ranges, etc.) or you want to generate a model that works very precisely in specific cases (for example for whole blood samples, measured on the 27k Illumina platform).

Conclusions¶

  • Know your data!
    • Check datafiles by manually looking into them.
    • Be very careful when parsing strings.
    • Explore your data before model building, check distributions, look for missing values, find outliers.
    • Use standardization in most cases.
  • Know your goal!
    • Do you want to build a model that generalizes to all cases "well enough"?
    • Or do you want to build a model that is extremely precise in a few well-defined cases?
  • Make sure your model makes sense!
    • Do not fit a model with many parameters to a dataset of a few samples. - Avoid overfitting!
    • Use regularization to reduce the chance of overfitting.
  • Be careful when interpreting predictive performance!
    • Make sure to test your models on unrelated, never-before-seen data. If you separate a test set from the original dataset and get good results, it is still possible that the model learned some technical artifact and not a real trend.
    • Always have a good understanding of the underlying data sources.
  • Sometimes additional data can increase model performance way more than clever coding tricks. Take the time and effort to collect and preprocess as much data as you can.
